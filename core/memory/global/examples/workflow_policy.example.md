# Workflow Policy

## Standard Flow
1. Explore
2. Plan -- before finalizing approach, invert: what would definitely make this fail? Eliminate those paths first.
3. Implement
4. Review
5. Handoff

## Project Memory
- `docs/REQUIREMENTS.md` defines what is being built.
- `docs/PLAN.md` defines staged execution.
- `docs/PROGRESS.md` records completed work and decisions.
- `docs/NEXT_STEPS.md` is the next-session handoff.
- `docs/RUN_CONTEXT.md` records runtime assumptions, execution profiles, API env vars.

## Bounded Automation
When running an unattended or semi-attended loop, record before starting:
- objective
- candidate set or input parameters
- max iterations or time limit
- evaluation rubric
- expected artifact outputs
- stop condition
- human review checkpoint

No unattended code-writing, auto-merge, or production-promotion loops without explicit approval.

## Cost Acknowledgment
Before any paid hosted-model run, estimate the run cost and ask:
> `acknowledge the cost and proceed? [y/n]`

## Publication Boundary
Safe to commit: `AGENTS.md`, `CLAUDE.md`, `docs/*.md`, shared runtime files, code, tests.
Keep local only: `.claude/settings.local.json`, `.env*`, `secrets/`, private keys.

## Review Gate
Review required before merge for logic changes, data pipeline changes, and shared artifact changes.

## Signal-over-Noise Rules
- State the highest-signal fact first. Do not bury the conclusion.
- If a finding changes nothing actionable, it is noise -- log it but do not escalate.
- Second-order effects check: before closing any decision, ask what happens after the immediate
  effect lands. Who is affected downstream? What does the system do next? If the second-order
  effect is worse than the first-order gain, the decision is not finished.
